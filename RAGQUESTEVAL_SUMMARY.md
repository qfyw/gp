# RAGQuestEval 指标总结

## 概述

**RAGQuestEval** 是 CRUD-RAG 论文提出的核心评估指标，用于评估 RAG 系统生成内容的质量。与传统基于字符串匹配的指标不同，RAGQuestEval 通过问答方式间接评估生成质量，更符合语义理解和实际应用场景。

## 核心指标

### Quest Avg F1

**定义**: 所有问答对 F1 分数的平均值

**计算方式**:
```python
F1 = 2 * (Precision * Recall) / (Precision + Recall)
Quest Avg F1 = mean(F1 for all QA pairs)
```

**意义**: 衡量生成文本中信息的准确性和完整性

### Quest Recall

**定义**: 1 - (无法推断的回答比例)

**计算方式**:
```python
Quest Recall = 1 - (count("无法推断") / total_answers)
```

**意义**: 衡量生成文本对原始信息的覆盖程度

## 工作原理

```
Ground Truth 文本
    ↓
[LLM] 提取关键信息 + 生成问题
    ↓
问题列表 [Q1, Q2, Q3, ...]
    ↓
    ├─→ [LLM] 用 Ground Truth 回答 → 参考答案 [A1_gt, A2_gt, A3_gt, ...]
    └─→ [LLM] 用生成文本回答 → 预测答案 [A1_pred, A2_pred, A3_pred, ...]
    ↓
[计算] F1 分数和 Recall
```

## 与其他指标对比

| 指标 | 评估方式 | 优势 | 局限性 |
|------|---------|------|--------|
| **RAGQuestEval** | 问答方式 | 语义理解，接近实际应用 | 成本高，需要 LLM |
| **BLEU** | N-gram 匹配 | 快速、标准化 | 不适合中文，语义弱 |
| **ROUGE** | N-gram 重叠 | 适合摘要任务 | 忽略语义，关键信息权重低 |
| **BERTScore** | 语义相似度 | 考虑语义 | 计算成本高 |
| **EM** | 完全匹配 | 严格准确 | 过于严格 |

```csv

```

## 测试结果

### 最新结果（20条数据）

| 指标 | 平均值 | 标准差 |
|------|--------|--------|
| Quest Avg F1 | 0.7254 | ±0.3193 |
| Quest Recall | 0.7171 | ±0.3445 |

### 表现分布

| 表现等级 | F1 范围 | 样本数 | 占比 |
|---------|---------|--------|------|
| 优秀 | ≥ 0.8 | 9 | 45% |
| 中等 | 0.4-0.8 | 5 | 25% |
| 较差 | < 0.4 | 6 | 30% |

### 典型案例

#### 优秀案例 (F1 = 1.0)

**问题**: 试验药物NP137针对什么疾病？
- 参考答案: 晚期子宫内膜癌
- 预测答案: 晚期子宫内膜癌 ✓

**问题**: 有多少名患者参与了人体试验？
- 参考答案: 9名
- 预测答案: 9名 ✓

#### 中等案例 (F1 = 0.25)

**问题**: 西安发放的体育类消费券金额是多少？
- 参考答案: 500万元
- 预测答案: 43万元 ✗ (数字错误)

**问题**: 检索到了错误的数据源（不同时间的消费券）

#### 失败案例 (F1 = 0)

**问题**: 哪些人群容易患上干眼症？
- 参考答案: 儿童青少年
- 预测答案: 无法推断 ✗

**问题**: 检索失败，未找到相关文档

## 主要问题分析

### 1. 数字准确性问题 (30% 样本)
- 不同时间/地点的数据混淆
- 需要更精确的实体和时间匹配

### 2. 检索失败 (15% 样本)
- 关键词召回不足
- 需要优化检索策略

### 3. 部分信息缺失 (20% 样本)
- 多子问只回答部分
- 需要改进提示词完整性

## 优化建议

### 优先级1: 优化实体和时间匹配
- 实现更精确的数字识别
- 添加时间范围过滤
- 区分不同来源的数据

### 优先级2: 改进检索召回
- 增加 top_k
- 优化关键词权重
- 提升检索多样性

### 优先级3: 提示词优化
- 强调完整性要求
- 增加多子问示例
- 改进回答格式约束

## 最佳实践

### 数据准备
1. **样本量**: 建议至少 10-20 条数据
2. **数据质量**: 确保原文和生成文本完整
3. **多样性**: 覆盖不同类型的问题和场景

### 成本控制
RAGQuestEval 需要多次调用 LLM（每条数据约 3-5 次）：

- **10 条数据**: 约 30-50 次 API 调用
- **50 条数据**: 约 150-250 次 API 调用

**建议**:
- 开发阶段使用小样本（5-10 条）
- 正式评测使用中等样本（20-30 条）
- 选择性保存问题和答案，避免重复生成

### 结果分析
1. **查看详细结果**: 分析哪些类型问题容易出错
2. **对比不同配置**: 量化优化效果
3. **结合其他指标**: 全面评估系统性能

## 故障排查

### 评估结果全为 0

**可能原因**:
- LLM API 配置错误
- 问题生成失败
- 所有答案都是"无法推断"

**解决方法**:
```bash
# 检查环境变量
echo $OPENAI_API_KEY
echo $OPENAI_API_BASE

# 查看 .env 文件
cat .env
```

### 评估速度很慢

**可能原因**:
- 每条数据需要多次 LLM 调用
- 网络延迟
- LLM 响应慢

**解决方法**:
- 减少样本量
- 使用更快的 LLM（如 qwen-flash）
- 增加超时时间

## 核心优势

1. **语义理解**: 不依赖严格字符串匹配
2. **实用性强**: 接近实际应用场景
3. **可解释性**: 通过问答对分析错误

## 主要局限

1. **成本高**: 需要多次 LLM 调用
2. **依赖 LLM**: 评估质量受 LLM 能力影响
3. **速度慢**: 不适合实时评估

## 适用建议

- **研发阶段**: 用于优化和调优
- **评测阶段**: 用于全面评估
- **生产环境**: 不建议用于实时评估
